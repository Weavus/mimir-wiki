# AGENTS.md - mimir-wiki

## Current Status

This repository is currently a specification/workspace for `mimir-wiki`, not an
implemented Python package yet.

Key files:

- `mimir_wiki_spec.md` - main implementation specification for `mimir-wiki`
- `mimir_workflow_goal_spec.md` - wider Mimir workflow and design context
- `cache/` - example `mimir-confluence` exports to use when validating
  assumptions about real input data

Do not assume `src/`, `tests/`, `pyproject.toml`, or CLI commands exist until
they have been created in this repo.

## Project Purpose

`mimir-wiki` consumes exported `mimir-confluence` caches and produces curated,
reviewable Markdown knowledge. It should keep raw source evidence separate from
generated knowledge.

Primary workflow:

```text
mimir-confluence cache
  -> validation and enrichment
  -> document/theme/concept/quality indexes
  -> Onyx POC enriched Markdown in dist/onyx-enriched
  -> enrichment reports
  -> later compiled Markdown in an Obsidian-compatible vault
```

## Real Sample Caches

Use `cache/` as the local reality check for the exporter schema and real
Confluence content shape. Current sample exports include:

- `cache/aaa-domain-application-support/` - broad mixed application/support
  export, 3517 successful pages in space `CEREE`
- `cache/carel3support/` - broad mixed support export, 828 successful pages in
  space `carel3support`
- `cache/customer-identity-and-access-management-entra/` - mixed product,
  support, runbook and design export, 695 successful pages in space `CIAME`
- `cache/carel3support-rca/` - RCA-oriented export, 526 successful pages in
  space `carel3support`

Do not overfit the implementation to RCA pages. The first MVP should enrich and
inventory the broader mixed exports, with RCA-specific classification signals
enabled when the selected cache contains RCA-heavy content. Wiki compilation is
post-MVP.

Observed layout:

```text
cache/{dataset_name}/
  dataset.json
  manifest.jsonl
  manifest.summary.json
  errors.jsonl              # optional; present when export/download errors occurred
  pages/{page_id}/
    metadata.json
    clean.md
    text.txt
    raw_storage.html
    raw_export_view.html
    links.json
    conversion.json
    attachments/
```

`errors.jsonl` is optional. It can exist even when `manifest.summary.json`
reports a complete export, so validation and reports should inspect it
separately.

Important observed contract details:

- `manifest.jsonl` entries are intentionally compact. They include fields such
  as `page_id`, `path`, `markdown_path`, `space_key`, `status`, `title`,
  `updated_at`, and `version`.
- Richer source metadata lives in each page's `metadata.json`.
- `metadata.json` uses `content_hashes` with separate hashes such as
  `markdown_sha256`, `text_sha256`, `storage_sha256`, and
  `export_view_sha256`.
- `author` is an object with `display_name` and `username`, not a plain string.
- `ancestors` use `id` and `title`, not `page_id`.
- `links.json` is a separate artifact and should be loaded for outbound link
  analysis instead of expecting links inside `metadata.json`.
- `conversion.json` records converter version, conversion timestamp, content
  hashes, and conversion warnings.
- `clean.md` includes YAML front matter generated by `mimir-confluence`; treat
  it as source evidence, not curated knowledge.

## Implementation Guidance

Start by making the enrichment and inventory workflow executable against the
observed cache before building LLM-heavy features or wiki compilation.

Recommended first modules once `src/` exists:

- cache reader/validator for the real `mimir-confluence` cache layout
- Pydantic models for manifest rows, page metadata, links, conversion records,
  enrichment output, document index rows, quality scores, themes, concepts and
  candidate entities
- config loader for `mimir-wiki.yaml`, `.env` and environment variables
- deterministic document classification and quality signals
- deterministic `--provider none` enrichment path for tests
- LLM provider interface with OpenAI, Azure OpenAI and Azure AI Foundry
  implementations for semantic enrichment
- task-specific model routing for classification, summaries, keywords, themes,
  concepts, candidate entities, operational signals and quality warnings
- general document classification, with RCA-oriented signals enabled when a
  cache contains RCA-heavy content
- Onyx POC Markdown export under `dist/onyx-enriched/{dataset_name}/` with
  required first-line `#ONYX_METADATA={...}` JSON containing `link`,
  `file_display_name`, and `doc_updated_at`
- enrichment reports that summarize document types, themes, stale/deprecated
  pages and high-value sources

Recommended package boundaries:

```text
src/mimir_wiki/
  cli.py
  config.py
  schemas.py
  pipeline.py
  cache_reader.py
  enrichers/
    deterministic.py
    llm.py
    prompts/
  llm/
    base.py
    openai_provider.py
    azure_openai_provider.py
    azure_foundry_provider.py
  scoring.py
  writers/
    enrichment_json.py
    jsonl_indexes.py
    onyx_markdown.py
  reports.py
```

Prompt templates should be versioned files. LLM response caches should include
source content hash, task, prompt version, provider and model/deployment.
Concurrency, timeouts, retries and rate limits should be config-driven.

All live LLM calls must go through one shared rate-limited client wrapper. It
must enforce configurable concurrency, optional requests-per-minute and
tokens-per-minute limits, per-request timeouts, bounded exponential backoff with
jitter, `Retry-After` handling, and a maximum retry count. Retry transient
failures such as timeouts, connection resets, HTTP 408/409/429/500/502/503/504
and provider overload/rate-limit errors. Do not retry missing credentials,
authorization failures, invalid requests, context-length failures that require
chunking changes, or schema validation failures after a syntactically valid
response. Final provider failures should be captured in enrichment artifacts
and reports unless fail-fast mode is configured.

Use bounded concurrency rather than ad hoc unbounded threading. Prefer async
I/O for provider calls, a shared LLM semaphore/rate limiter, bounded page worker
queues, and deterministic single-writer paths for output files. Thread pools are
acceptable for blocking file/HTML parsing only if profiling shows they help.
Expose `page_workers`, `llm_workers`, and `writer_workers` in config.

Do not commit secrets. OpenAI, Azure OpenAI and Azure AI Foundry keys,
endpoints, API versions and deployment names should come from environment
variables or `.env`.

Keep source cache files read-only from `mimir-wiki` except for writing derived
artifacts such as `pages/{page_id}/enrichment.json` when the user chooses that
output mode.

Prefer writing generated global artifacts outside the cache:

```text
knowledge/
dist/onyx-enriched/
reports/
runs/
```

## MVP Implementation Contract

Treat `mimir_wiki_spec.md` section 21.4 as the implementation contract for MVP
1. Do not start by inventing looser ad hoc output formats.

Required implementation details:

- Add `schema_version: mimir-wiki/v1` to generated JSON and JSONL rows. For
  Onyx POC Markdown, keep high-cardinality implementation metadata such as
  schema version and run ID in the Markdown body rather than the first-line
  Onyx metadata JSON.
- Create one `run_id` per CLI execution and write run artifacts under
  `runs/{run_id}/`, including `summary.json`, `page_failures.jsonl`,
  `warnings.jsonl` and `llm_usage.jsonl` when applicable.
- Keep global JSONL outputs stable and diffable by sorting rows, preferably by
  `(space_key, page_id)` or a documented stable key.
- Use atomic writes for generated files.
- Implement config precedence as: built-in defaults, config file, profile,
  `.env`, environment variables, CLI flags.
- Control durable feature defaults in `mimir-wiki.yaml` under `features`.
  CLI flags should override a single run for experiments. Write the resolved
  feature set into `runs/{run_id}/summary.json`.
- Use `--provider none` to force live LLM features off. Use `--enable-llm`,
  `--disable-llm`, and repeatable `--llm-task TASK` for coarse runtime
  overrides rather than adding many one-off switches.
- Make `--changed-only` compare source content hash plus schema version, prompt
  version, provider, model/deployment, task and enrichment config hash.
- Keep `keywords`, `themes`, `concepts` and `candidate_entities` semantically
  distinct, with confidence, method and source evidence.
- Do not parse binary attachments in MVP 1. Index attachment names/counts only
  and report pages with attachments as follow-up candidates.
- Add configurable chunking for oversized LLM inputs. Context-length failures
  should be non-retryable and should recommend chunking/config changes.
- Run generated Onyx POC Markdown through configurable redaction rules before
  writing when redaction is enabled.
- Do not use YAML front matter for Onyx POC Markdown. The Onyx file connector
  metadata must be the first line of the file as `#ONYX_METADATA={...}`.
  Keep this first-line metadata lean: required Onyx fields plus low-cardinality
  filter tags. Put high-cardinality values such as `run_id`, `document_id`,
  `page_id`, source hashes and prompt versions in the Markdown body instead.
- Record provider/model/task token usage for live LLM calls when providers
  return usage data.

## CLI Intent

The planned CLI is `mimir-wiki` implemented with Typer.

Initial commands from the spec:

- `validate-cache`
- `enrich`
- `report`

Defer `extract-facts`, `build-entities`, `rca-index`, all `compile ...`

## CLI UX Conventions

The CLI should be human-friendly by default and scriptable when requested. Use
Typer for command structure and Rich for colour, tables, panels and progress.

Apply these conventions consistently:

- Reuse common options across commands: `--config`, `--profile`, `--cache`,
  `--out`, `--provider`, `--limit`, `--changed-only`, `--force`, `--dry-run`,
  `--json`, `--no-color`, `--quiet`, `--verbose`, and `--log-file`.
- Show colour and progress bars only when they help. Default to auto-detection:
  enable them for interactive TTYs, disable them for `--json`, `--quiet`, CI and
  redirected output.
- Use semantic colour: green for success, yellow for warnings or partial
  success, red for errors, cyan or blue for paths and command names, and dim
  styling for low-priority detail. Do not make colour the only status signal.
- Every long-running command should show bounded progress for pages considered,
  pages processed, skipped pages, failed pages and LLM retries when applicable.
- Every command should end with a concise summary unless `--json` or `--quiet`
  is set. Include dataset, cache path, pages processed/skipped/failed, files
  written, reports written, provider/model usage, retry count, warning count and
  elapsed time where relevant.
- `--dry-run` must validate and plan without writing files, then show planned
  page counts and output paths.
- Normal errors should explain what failed, which page/path was involved and
  what to try next. Reserve stack traces for `--verbose`.
- Keep exit codes predictable: `0` success, `1` validation/user input problems,
  `2` provider/runtime failure after retries, `3` partial success with recorded
  page-level failures.

Prefer testing CLI behavior through semantic assertions on exit codes, emitted
summary fields, JSON output and generated artifacts rather than brittle exact
colour snapshots.

## Testing Conventions

When tests are added:

- Prefer focused MVP tests around cache parsing, schema validation, document
  indexing, enrichment output, scoring, changed-only behavior and report
  generation. Also test that Onyx POC Markdown starts with first-line
  `#ONYX_METADATA={...}` JSON containing `link`, `file_display_name`, and
  `doc_updated_at`. Add approved-publish filtering tests when the
  compilation/publish phases begin.
- Use fixtures derived from the mixed exports under `cache/`, but keep test
  fixtures minimal and sanitized.
- Include tests for the real observed schema differences listed above.
- Include deterministic tests for `--provider none`; no test should require a
  live Confluence instance or live LLM provider.

## Repo-Specific Conventions

- Treat `mimir_wiki_spec.md` as the current source of design intent, but verify
  claims against the real exports under `cache/`.
- Keep raw source evidence separate from curated Markdown outputs.
- Do not publish or approve AI-generated pages automatically.
- Every major compiled factual section should carry source evidence or an
  explicit `unknown`, `not found`, or `needs review` marker.
- RCA documents are historical evidence. They are strong evidence for impact,
  timeline, root cause, contributing factors, and lessons, but weak evidence
  for current operating procedure unless corroborated by current runbooks or
  support documentation.
- General Confluence exports may contain mixed application, architecture,
  runbook, support, project, and reference pages. Keep classifiers and source
  ranking generic, then specialize scoring by document type and claim type.
- Use standard Markdown links initially; Obsidian wikilinks can be an optional
  later render mode.
- Use ASCII in files unless an existing file or source artifact requires
  otherwise.
